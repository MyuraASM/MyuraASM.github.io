PussyOS

v1.0 Documentation

A custom x86-64 operating system built from scratch with a unique aesthetic and personality. Features include a custom bootloader, filesystem, login system, and terminal interface.

Overview

PussyOS is a minimal x86-64 operating system that boots into long mode and provides a basic interactive shell environment. The OS features custom boot animations, a login system with configuration file support, and a persistent filesystem.

Key Features

  • Custom Multiboot2-compliant bootloader
  • Long mode (64-bit) kernel with paging
  • VGA text mode display with color support
  • Interrupt Descriptor Table (IDT) for keyboard input
  • Simple filesystem with disk persistence
  • User authentication system with configurable credentials
  • Animated boot sequence with ASCII art
  • Terminal UI with command processing

Boot Sequence

The boot process follows these stages to transition from BIOS/UEFI to a running 64-bit kernel:

1. Multiboot Header

The OS uses a Multiboot2 header to be loaded by GRUB or compatible bootloaders: 

section .multiboot_header
header_start:
    dd 0xe85250d6    ; Multiboot2 magic number
    dd 0             ; Protected mode i386
    dd header_end - header_start
    dd 0x100000000 - (0xe85250d6 + 0 + (header_end - header_start))
    
    ; End tag
    dw 0
    dw 0
    dd 8
header_end:

2. 32-bit Protected Mode Entry

The bootloader enters at the start label in 32-bit protected mode. The kernel performs several validation checks:

  • Multiboot check: Verifies EAX contains the magic number 0x36d76289
  • CPUID check: Tests if CPUID instruction is available
  • Long mode check: Ensures CPU supports 64-bit mode via CPUID

3. Page Table Setup

Before enabling long mode, identity-mapped page tables are configured: 

setup_page_tables:
    ; Link L4 -> L3
    mov eax, page_table_l3
    or eax, 0b11    ; Present + Writable
    mov [page_table_l4], eax

    ; Link L3 -> L2
    mov eax, page_table_l2
    or eax, 0b11
    mov [page_table_l3], eax

    ; Map first 1GB (512 * 2MB pages)
    mov ecx, 0
.loop:
    mov eax, 0x200000
    mul ecx
    or eax, 0b10000011  ; Present + Writable + Huge
    mov [page_table_l2 + ecx * 8], eax
    inc ecx
    cmp ecx, 512
    jne .loop
    ret

Why Identity Mapping?

Identity mapping means virtual addresses equal physical addresses. This simplifies the initial boot process since the kernel can execute without address translation confusion.

4. Enabling Paging & Long Mode

enable_paging:
    ; Load CR3 with PML4 address
    mov eax, page_table_l4
    mov cr3, eax

    ; Enable PAE (Physical Address Extension)
    mov eax, cr4
    or eax, 1 << 5
    mov cr4, eax

    ; Set long mode bit in EFER MSR
    mov ecx, 0xC0000080
    rdmsr
    or eax, 1 << 8
    wrmsr

    ; Enable paging in CR0
    mov eax, cr0
    or eax, 1 << 31
    mov cr0, eax
    ret

5. Jump to Long Mode

After loading the GDT64 with a 64-bit code segment, the kernel performs a far jump to transition into 64-bit long mode: 

lgdt [gdt64.pointer]
jmp gdt64.code_segment:long_mode_start

6. Long Mode Entry Point

bits 64
long_mode_start:
    ; Zero out segment registers
    mov ax, 0
    mov ss, ax
    mov ds, ax
    mov es, ax
    mov fs, ax
    mov gs, ax

    ; Call C kernel
    call kernel_main
    hlt

Kernel Initialization

Once in 64-bit mode, the kernel_main function initializes all subsystems:

Initialization Sequence

void kernel_main(void) {
    // 1. Initialize display
    print_clear();
    print_set_color(PRINT_COLOR_WHITE, PRINT_COLOR_BLACK);

    // 2. Initialize disk
    disk_init();

    // 3. Load or create filesystem
    if (load_fs(100) != 0) {
        // Start fresh if no saved FS
        print_str("No saved filesystem found. Starting fresh.");
    } else {
        print_str("Filesystem loaded from disk.");
    }

    // 4. Initialize IDT and keyboard
    idt_init();

    // 5. Show welcome screen with login
    show_welcome_screen();

    // 6. Enter main loop
    while (1) {
        __asm__ volatile("hlt");
    }
}

Subsystem Details

Subsystem Function Purpose
Display print_clear() Initialize VGA text mode buffer at 0xB8000
Disk disk_init() Initialize ATA/IDE disk driver
Filesystem load_fs(100) Load filesystem from LBA sector 100
Interrupts idt_init() Set up IDT and enable keyboard interrupts
UI show_welcome_screen() Display ASCII art and login prompt

Login System

PussyOS features a complete authentication system with configurable credentials stored in the filesystem.

Configuration File Format

Login credentials are stored in a file named config in the filesystem: 

username. admin
password. pass

Security Note

The current implementation stores passwords in plaintext. This is suitable for a learning/hobbyist OS but would require hashing (e.g., SHA-256) for any real-world use.

Login Flow

  1. Load config: Read credentials from config file (defaults to admin/pass if missing)
  2. Username prompt: User enters username
  3. Password prompt: User enters password (visible input)
  4. Validation: Compare inputs against stored credentials
  5. Success: Display boot animation and enter shell
  6. Failure: Decrement attempt counter, lock system after 3 failures

Credential Parsing

The load_credentials() function parses the config file with these features:

  • Strips leading and trailing whitespace
  • Expects format: username. <value> and password. <value>
  • Falls back to defaults if file is missing or malformed
  • Creates config file with defaults if it doesn't exist

Attempt Limiting

Users get 3 login attempts. After 3 failures, the system enters a locked state: 

if (login_attempts > 0) {
    print_str("Invalid credentials. Attempts remaining: ");
    print_int(login_attempts);
} else {
    print_str("Too many failed attempts. System locked.");
    system_locked = 1;
    while (1) {
        __asm__ volatile("hlt");
    }
}

Typing Effect

Text appears character-by-character with a calibrated delay: 

void type_text(char* text, uint8_t fg_color, uint8_t bg_color) {
    print_set_color(fg_color, bg_color);
    for (size_t i = 0;; i++) {
        char ch = text[i];
        if (ch == '\0') break;
        print_char(ch);
        delay(2);  // Tuned for smooth typing speed
    }
}

Progress Bar

An animated progress bar shows system initialization: 

void animate_progress() {
    print_str("Initializing system... [");
    for (int progress = 0; progress <= 40; progress++) {
        // Draw filled portion
        for (int j = 0; j < progress; j++) print_char('=');
        // Draw empty portion
        for (int j = progress; j < 40; j++) print_char(' ');
        print_char(']');
        print_char('\r');  // Carriage return to overwrite
        
        // Color pulse effect
        uint8_t fg = (progress % 10 < 5) ? 
            PRINT_COLOR_MAGENTA : PRINT_COLOR_RED;
        print_set_color(fg, PRINT_COLOR_BLACK);
        
        delay(1);
    }
}

Delay Calibration

The timing system uses a busy-loop with NOP instructions: 

void delay(uint32_t units) {
    for (uint32_t i = 0; i < units * 100000; i++) {
        __asm__ volatile("nop");
    }
}

The multiplier (100000) is calibrated for typical x86-64 clock speeds to produce visually pleasing animation timing.
